# US-022: Shared-task behavior on dev surface and dashboard - Piano di Implementazione

**Generato da:** Archetipo Planning Team
**Data:** 2026-05-06

---

## User Story

**Epic:** EP-001 - Sprint & Backlog Management
**Priorità:** MEDIUM | **Story Points:** 3

**Story**
As a developer co-assigned to a shared task, I want the task to appear in my daily view, to share the morning commitment with the other co-assignees, and to submit my own check-in and end-of-day retro independently, so that we collaborate on the same task without losing individual visibility on each dev's state.

**Criteri di Accettazione**
- [ ] A shared task appears in the today's-tasks list (US-007) of every co-assignee
- [ ] Commitment is shared: when one co-assignee adds the shared task to today's plan, the task counts as committed for all co-assignees and appears in their "today's plan" automatically (extends US-008)
- [ ] Mid-day check-in is per-dev: each co-assignee submits their own status (On Track / Blocked / Task Changed) on the shared task, independently of the others (extends US-009)
- [ ] End-of-day retro is per-dev: each co-assignee submits their own outcome (Done / In Progress / Not Started) on the shared task, independently of the others (extends US-010)
- [ ] Silent auto-flag is per-dev on shared tasks: a co-assignee who has not personally responded is flagged Silent regardless of other co-assignees' responses (extends US-011)
- [ ] On the team dashboard (US-012), a shared task appears in the row of every co-assignee, marked with a "share" icon to distinguish it from single-assignee tasks; each row reflects that dev's own check-in/retro state on the shared task
- [ ] Depends on US-021 (multi-assignment must exist in the data model before this behavior can be exercised)

## Soluzione Tecnica

US-022 non introduce nuove pagine: definisce il **contratto dati** per le entità di engagement giornaliero (commitment, check-in, retro, silent flag) in modo che US-008..US-012 le adottino già con la semantica corretta - shared per il commitment, per-dev per tutto il resto. La share-ness di un task è una proprietà derivata (`task.assignees.length > 1`), non un flag separato: questo evita stati incoerenti tra "task con un solo assegnatario" e "task condiviso ridotto a uno". L'icona "share" nel dashboard e i chip dei co-assegnatari sul dev page sono pure overlay visuali sopra i componenti già pianificati. Poiché US-008-12 non sono ancora implementate, questo piano definisce le shape dati canoniche e i comportamenti UI come **vincoli** che quelle storie dovranno rispettare; introduce inoltre una piccola libreria di helper `src/lib/sharing.ts` riutilizzabile da tutte le storie a valle.

- **Commitment task-keyed (`prisma/schema.prisma`)**: nuovo modello `DailyCommitment { taskId, sprintDay (Date), createdById (devProfileId di chi ha committato per primo), createdAt; @@id([taskId, sprintDay]) }`. Niente `devProfileId` nella PK: il commitment esiste o non esiste per il task in quel giorno - tutti i co-assignee lo "ereditano". `createdById` è informativo (audit/dashboard "committed by"), non chiave.
- **Engagement per-dev (`prisma/schema.prisma`)**: tre tabelle distinte con compound key `(taskId, devProfileId, sprintDay)`:
  - `TaskCheckIn { taskId, devProfileId, sprintDay, status (enum OnTrack|Blocked|TaskChanged), note (String?), submittedAt }`
  - `TaskRetro { taskId, devProfileId, sprintDay, outcome (enum Done|InProgress|NotStarted), submittedAt }`
  - `TaskSilentFlag { taskId, devProfileId, sprintDay, flaggedAt }` (popolata dal job di US-011 quando il dev non risponde entro 2h)
  - Tutte con `@@id([taskId, devProfileId, sprintDay])` e FK `Restrict` su `devProfile` (preserva storico anche post-soft-delete) e `Cascade` su `task`. Indice secondario `(devProfileId, sprintDay)` per le query del dev page e del dashboard.
- **Helper di derivazione (`src/lib/sharing.ts`)**: funzioni pure
  - `isShared(task: { assignees: unknown[] }) => boolean`: `assignees.length > 1`
  - `coAssignees(task, currentDevId)`: ritorna gli assignee diversi da `currentDevId` (per i chip "with X, Y" sul card del dev)
  - `engagementKey(taskId, devProfileId, sprintDay)`: chiave canonica per le tabelle per-dev
  - Tutte testabili in isolamento, senza Prisma.
- **Auto-inclusione del task condiviso nel "today's plan" (US-008)**: il caricamento del piano del giorno per un dev fa `findMany` su `DailyCommitment` `where: { sprintDay: today, task: { assignees: { some: { devProfileId: currentDev.id } } } }`. Così i task committati da un co-assegnatario compaiono automaticamente; il vincolo che un task possa essere committato solo se include il dev fra gli assignee è enforced lato server action.
- **Card del task sul dev page (estende US-007)**: quando `isShared(task)` è vero, il `TodayTaskCard` mostra una riga aggiuntiva "Shared with: <chip>, <chip>" (nomi dei `coAssignees`). Riusa il chip già introdotto da US-021 (`TaskRow` chips). Un'icona share inline (Lucide `Users` o equivalente già nel design system) accanto al titolo.
- **Stato per-dev su check-in e retro (estende US-009/010)**: le rispettive form action scrivono `TaskCheckIn` / `TaskRetro` con `devProfileId = currentDev.id` (mai con un dev diverso). Submit di un dev non altera lo stato dell'altro. Le query per la summary mostrata sul dev page filtrano per `devProfileId = currentDev.id`.
- **Silent flag per-dev (estende US-011)**: il cron/job che valuta i silent dev itera su `(devProfileId, taskId)` per i task committati: se non esiste un `TaskCheckIn` per quel triplo entro `midDayTime + 2h`, inserisce un `TaskSilentFlag`. Il job non assume single-assignee.
- **Dashboard (estende US-012)**: una riga per dev attivo. Per ogni riga, le sue assegnazioni del giorno; per ciascun task condiviso (`isShared`) renderizza l'icona share accanto al titolo del task. Lo stato visualizzato (On Track / Blocked / Silent / Done...) è quello del `(taskId, devProfileId)` di QUELLA riga - mai un'aggregazione fra co-assegnatari.
- **Nessuna nuova dipendenza**: tutto si appoggia a Prisma, Zod, design token esistenti, lucide-react (`Users` icon).
- **Nessun migration di backfill**: le tabelle sono nuove, US-008-12 le useranno da subito. Se invece US-008-12 atterrano PRIMA di US-022 con tabelle differenti (es. `Commitment(taskId, devProfileId)` per-dev), questo piano richiederà un task di migrazione aggiuntivo - segnalato come rischio in `Note di sequencing`.

**Note di sequencing**: US-008..US-012 non sono ancora pianificate. Questo piano è **prescrittivo**: chi pianificherà quelle storie deve adottare le shape dati definite qui (`DailyCommitment` task-keyed, `TaskCheckIn/Retro/SilentFlag` per-dev). Se ciò non avviene, US-022 dovrà includere una migrazione per ricostruire le tabelle, che oggi non è necessaria.

## Strategia di Test

I test si concentrano sugli invarianti di sharing (commitment condiviso, stato per-dev isolato) e sulla derivazione `isShared`, perché sono il cuore non banale di questa storia. UI e flussi end-to-end sono test deferred coerentemente con US-006/US-021 (no harness Playwright). Tutta la copertura significativa è ottenibile in unit/integration sulle entità del data model e sugli helper.

- **Unit (`src/lib/sharing.test.ts`)**: `isShared` ritorna `false` per 1 assignee, `true` per 2+, `false` per array vuoto (caso teorico, non dovrebbe accadere ma deve essere safe). `coAssignees` esclude correttamente l'id corrente. `engagementKey` produce chiavi deterministiche.
- **Integration (sql-level, su DB di test se disponibile, altrimenti documentato come deferred)**: inserire `DailyCommitment` per un task con 2 assignee; verificare che la query "today's plan" del dev A e del dev B ritornino entrambi quel task. Inserire `TaskCheckIn` per dev A: la query "summary" del dev B non lo ritorna. Tentare di inserire un secondo `DailyCommitment` per lo stesso `(taskId, sprintDay)`: deve fallire per violazione PK.
- **Integration (logica di share su task con un solo assignee)**: garantire che le query e gli helper si comportano identici al caso non condiviso quando `assignees.length === 1` (no regressione su single-assignee).
- **DEFERRED (E2E)**: niente Playwright nel progetto.
- **DEFERRED (DB integration)**: già deferred trasversalmente in US-003/US-021 per assenza di harness DB. Sostituito da unit test sugli helper puri e da review del Prisma schema.
- **Smoke check post-merge**: `npx tsc --noEmit` e `npx tsx --test src/lib/**/*.test.ts` devono passare; verifica manuale che `prisma generate` produca client coerente con i tre nuovi modelli.
